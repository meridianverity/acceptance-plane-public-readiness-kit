# Public Surface Alignment

Boundary: Not a product implementation. Not a standard. No patent license. No certification right.

The public readiness kit should stay aligned with the current Meridian Verity public surface while remaining more conservative than a marketing page.

## Stable public-language anchors

Use these as alignment anchors when preparing release notes, repository summaries, and reviewer-facing summaries:

- Proof before impact.
- Receipts, not promises.
- Bring one protected action boundary before asking for private depth.
- No confidential implementation, customer, regulated, legal, security-sensitive, or deployment-specific material before written scope.
- Missing, stale, mismatched, or unreplayable proof should not become approval.
- Public materials do not create certification, conformance approval, implementation rights, patent licenses, or production deployment rights.

## Release-note discipline

A public release note may say that the kit helps teams discuss action acceptance before impact. It must not say that the kit is a product implementation, formal standard, conformance program, certification kit, production verifier, exact evidence schema, cryptographic binding method, patent claim chart, or deployment approval path.

Before announcing a public release, compare the release body, repository description, and public website for obvious mismatch. If they differ, prefer the narrower boundary language until counsel, security, and release owners agree on updated wording.
