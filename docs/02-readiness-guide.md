# Readiness Guide

This guide helps teams determine whether they can reason about autonomous action acceptance before impact. It does not certify readiness.

## Readiness questions

1. Can the team name the protected systems affected by autonomous actions?
2. Can the team distinguish output generation from action commitment?
3. Can the team identify the acceptance boundary for each high-consequence action?
4. Can the team list what evidence must be current at the moment of action?
5. Can the team explain when an action should be ACCEPTED, HELD, or REFUSED?
6. Can the team produce a human-understandable reason for each decision?
7. Can the team avoid relying only on post-hoc logs?
8. Can the team prevent stale authority, wrong target, wrong scope, replay, or revocation ambiguity from becoming real-world harm?
9. Can the team explain how protected execution differs from accepted action?
10. Can the team prove to a buyer, auditor, regulator, or affected person why an action was accepted?

## Maturity ladder

| Level | Description |
| --- | --- |
| L0 — Unnamed | Autonomous actions are discussed as generic permissions, outputs, or logs. |
| L1 — Named | High-consequence actions and protected systems are identified. |
| L2 — Boundary-aware | Acceptance boundaries are documented before deployment. |
| L3 — Evidence-aware | Gate-time evidence categories are identified for each action class. |
| L4 — Decision-aware | ACCEPT / HOLD / REFUSE paths and human escalation are defined. |
| L5 — Verifier-ready posture | The organization can explain why action acceptance is evidence-bound, time-bound, and scope-bound, while using licensed/approved mechanisms for production enforcement. |

This ladder is educational. It is not a certification scale.
